Onereach.ai Lite

A slim desktop kernel that signs you in to Onereach.ai once and gives you a single home for the AIs you actually use. Lite is intentionally small: sign-in, your AI roster, a tools menu, articles, tutorials, and bug reporting — with everything else loaded on demand or deferred to the full app.

This guide walks through every feature in the order you’ll meet them. If you only have a minute, the Getting Started section gets you signed in and your first AI added.

Getting Started

On first launch you’ll see a small placeholder window with a Sign in button. Three steps from there to a working install:

  1. Sign in to Onereach. Click Sign in. An auth window opens at auth.<env>.onereach.ai; type your email and password as you would in a browser.
  2. Save a 2FA secret (optional but recommended). Open Onereach.ai Lite → Settings… → Two-Factor and scan or paste your TOTP secret. Lite generates the 6-digit code itself and auto-fills it the next time OneReach asks — no copy/paste from your phone. See Two-Factor.
  3. Add an AI key. Open Settings… → AI and paste an OpenAI / Anthropic key (or whichever providers you use). Without a key, the AI features show a banner pointing to this section.

Lite reuses the OneReach session you sign in to, so you don’t need to re-authenticate every time you open the app. The auth token never leaves the main process — renderers only see session metadata.

Signing In & Out

Sign-in flow

Clicking Sign in opens a separate auth window in its own session partition. Email and password are entered there as on any web login. When OneReach prompts for a 2FA code, Lite fills it for you if a TOTP secret is saved (otherwise you type the code from your phone). Once both authentication cookies (mult and or) are captured, the auth window closes and Lite is ready.

Sign-in completes typically within a few seconds. If something stalls, close the auth window and try again — nothing is partially persisted until both cookies arrive.

Sign out

Click Sign out in the placeholder. Lite clears in-memory tokens, removes every OneReach cookie from its partition, and deletes the persisted session record. The next launch starts fully signed-out.

If your session expires

OneReach sessions can expire on the server side. When Lite detects an expired session, it shows a re-sign-in prompt. Accept it to repeat the sign-in flow without losing local state (TOTP secrets, AI keys, IDWs all stay).

Account picker

If your OneReach login has access to multiple accounts, the auth window shows an account picker. If a target account id is set (via the IDW you opened from), Lite auto-selects it; otherwise pick manually.

Two-Factor (TOTP)

Lite includes a built-in TOTP authenticator. Once you save your OneReach 2FA secret, Lite both shows the live 6-digit code (so you can read it like Google Authenticator) and auto-fills it whenever the OneReach 2FA prompt appears.

Adding a secret

  1. Open Settings… → Two-Factor.
  2. Either scan the QR code on your OneReach 2FA setup page or paste the otpauth:// URI / Base32 secret.
  3. Lite confirms the secret matches by showing the current code; the same code on OneReach’s screen means it’s working.

Secrets live in your platform keychain (Keychain Access on macOS). They never leave your machine and are never logged. The 6-digit code itself is never logged either — only the fact that an attempt was made.

Auto-fill

When the OneReach auth window shows the 2FA prompt, Lite injects a MutationObserver that waits for the input to render, fills the current code, and clicks the verify button. If the secret isn’t saved yet, you’ll see a banner pointing to Settings instead.

Multiple secrets

If you have multiple OneReach accounts with separate 2FA secrets, save them in the same Two-Factor section. Lite picks the right one based on the account id in the URL of the prompt.

Settings

Open Onereach.ai Lite → Settings… from the menu (no shortcut by default; see Keyboard Input). The window is sectioned:

  • Account. Current sign-in info, sign-out, switch environment.
  • Two-Factor. TOTP secret management (see above).
  • AI. API keys for OpenAI, Anthropic, etc. Stored in the keychain.
  • IDWs. Manage your AI roster — presets and custom URLs (see Your AI Roster).
  • OAGI. Connection settings for the OAGI / Neon graph backend used for some Onereach features.
  • Updates. Auto-update behavior summary, channel, link to the releases page.
  • Diagnostics. Quick links to log files and the /logs stream — useful when reporting bugs.
  • Developer. Opens the API Reference window listing every documented Lite module + its events. Hidden behind a small expander.
  • About. Version, build info, source link.

The Settings window remembers which section you opened last. Some other parts of the app deep-link directly to a section (for example, the “needs setup” banner deep-links to Two-Factor).

Your AI Roster (IDWs)

An IDW (“intelligent digital worker”) is any AI you call up by name in Lite. The roster holds the AIs you use most so they’re one click away. Lite ships with quick-add presets for ChatGPT, Claude, Gemini, Perplexity, and Grok — click any of them in Settings → IDWs → Add and they appear in your roster immediately. You can also add a fully custom IDW by URL.

Adding an IDW

  1. Open Settings → IDWs.
  2. Click Add. The form shows quick-add buttons for the common ones; click a button to fill the form, or pick a kind manually and fill the URL.
  3. Save. The IDW appears as a row in your roster and in the catalog.

The catalog

Each IDW opens in its own browser window when clicked from the roster or catalog. Cookies and local storage stay isolated per IDW so signing in to ChatGPT doesn’t affect your Claude session. Some IDWs (those marked OneReach-aware) automatically ride your OneReach session.

Editing & removing

In Settings → IDWs, each row has an inline edit form. Removal asks for a confirmation; the confirmation is animated to give you a half-beat to change your mind.

Tools

The Tools menu is a user-curated list of label + URL shortcuts — think bookmarks tailored for the work you do here. Open Tools → Manage Tools… to add, rename, reorder, or remove entries. Each entry becomes a top-level item in the Tools menu.

Tools open in your default browser, not inside Lite. They’re for quick access to dashboards, docs, or external apps you reference often.

AI Run Times

AI Run Times is Lite’s built-in feed reader for AI news, announcements, and articles. It pulls from RSS-like sources and presents them in a tile grid; clicking a tile reads the article in a focused view, with optional text-to-speech for hands-free listening.

Adding sources

Open AI Run Times from the dock or the main window. Use the Add source control to point Lite at a feed URL; presets cover the common AI news sites.

Reading

Click a tile to open the article. The body fetches on demand from the source. The TTS button reads the article aloud using the system voice; the spoken audio is cached locally so re-reads are instant.

University

University is a curated list of short tutorials covering Lite features. Open it from the main window. Each tutorial is a single scroll — no quizzes, no logins, just the basics of a feature with screenshots.

New tutorials are added with releases. The list updates when you relaunch Lite after an update.

Bug Reports

Help → Report a Bug… opens a small modal. Fill in what happened, what you expected, and any reproduction steps. Lite automatically attaches diagnostic context:

  • The Lite version and build hash.
  • Recent log lines from the local log queue (port 47392).
  • Recent structured events (sign-in attempts, IPC calls, etc.).
  • OS / Electron version metadata.

Sensitive values are redacted before attachment: API keys (OpenAI, AWS, GitHub), JWTs, bearer tokens, and known credential patterns are replaced with [REDACTED:KIND]. TOTP codes and cookies were never logged in the first place. You can review and edit the attached payload before submitting.

Submitted reports go to the Onereach support queue.

Auto-Update

Lite checks for updates every 6 hours in the background and on demand via Help → Check for Updates…. Updates are distributed via GitHub releases at wilsr7000/Onereach_Lite_Desktop_App.

How an update flows

  1. Lite checks the GitHub release feed.
  2. If a newer version is available, you’re asked whether to download. The download itself doesn’t install — it stages the new version.
  3. You’re asked whether to Install and Restart now or defer until quit. Either way, the next launch runs the new version.
  4. On the next launch, Lite verifies the install actually took effect. If the version on disk doesn’t match what was downloaded, you get a dialog explaining the failure and offering a manual download.

If a version repeatedly fails to install

After three consecutive failures on the same version, Lite stops offering automatic install for that version and steers you to manual download. The next different version released retries auto-update normally. This keeps a single broken release from re-prompting you forever.

Dev builds

If you’re running Lite from source (e.g. npm start), Check for Updates… shows a dialog explaining that updates only run in packaged builds. Real checks happen against packaged installs.

Logs & Diagnostics

Lite runs a local log server on port 47392. Every module writes structured log lines and events there:

  • Logs. Level + category + message lines (e.g. [info] auth: hydrate complete).
  • Events. Typed structured records with names like auth.signIn.start, updater.check.finish, kv.set.fail. Each event carries elapsed duration for spans, plus operation-specific data.

Viewing logs

Open Settings → Diagnostics for a quick browser link to the local log server. The endpoint streams via WebSocket so logs update live. Bug reports automatically grab the last few hundred recent entries.

Categories

Each log line carries a category — auth, kv, updater, idw, tools, university, ai-run-times, neon, bug-report, menu, main-lite, logging. Filter by category in the log viewer to focus on one subsystem.

Keyboard Input

Lite ships with no keyboard shortcuts by default (per ADR-015). Menus exist for discoverability, not muscle memory. The thinking: shortcuts are personal, and an opinionated set baked in usually clashes with whatever the user already has.

The single exception is the Edit menu. Cmd+C / Cmd+V / Cmd+X / Cmd+Z / Cmd+A and their Ctrl-equivalents on Windows / Linux are role-driven and dispatch automatically to whichever input has focus. Without these, copy-paste wouldn’t work in Settings forms, agent windows, or the Bug Report modal — so the Edit menu items keep their platform-default shortcuts.

To add custom shortcuts: edit ~/.config/Onereach.ai Lite/Preferences manually, or wait for the planned keybindings UI in a future release.

Privacy & Security

Lite is designed so secrets stay local and never appear in logs or telemetry.

  • OneReach session tokens live in the auth window’s partition (cookie jar) and main-process memory. The renderer never sees the token directly — only session metadata (accountId, email).
  • API keys (OpenAI, Anthropic, etc.) live in your platform keychain via keytar. They are never written to disk in plaintext.
  • TOTP secrets live in the keychain. Lite generates the 6-digit code from the secret in main-process memory; the code and the secret are never logged.
  • Per-IDW partitions isolate each AI’s cookies and storage. Signing in to one IDW doesn’t affect others.
  • Bug-report redaction scrubs known credential patterns from log attachments before submission. You also see the attached payload before sending.

Code signing

Current Lite builds ship with a signed-but-not-notarized bundle, same posture as the full Onereach.ai app. On first install you see a one-time “Apple cannot check this app” warning; right-click the app and choose Open to bypass it. Auto-update works without prompts after that. Notarization is on the pre-launch roadmap (ADR-029).

Troubleshooting

The app shows “Apple cannot check” on first launch

Right-click the app in /Applications and choose Open. Confirm the second prompt. macOS remembers your choice; subsequent launches start normally and auto-updates work without further prompts.

Sign-in stalls on the 2FA page

Two common causes: (1) no TOTP secret saved (check Settings → Two-Factor); (2) account picker page that wasn’t auto-selected (just click the right account). If neither applies, check the log stream for auth.totp.* events.

Cmd+C / Cmd+V doesn’t work

That means the Edit menu isn’t built — usually a sign of a broken kernel boot. Check the log for menu-related errors at startup.

Auto-update never finds a new version

Confirm Help → Check for Updates… opens a real check (in dev mode it shows a “dev build” dialog instead). Confirm the GitHub releases page is reachable from your network. Logs under category updater show the response headers.

An IDW won’t open / shows blank

IDW windows are isolated per partition. If the AI’s site has changed its login flow, you may need to delete the IDW and re-add it so cookies start fresh. Settings → IDWs → remove + add again.

The placeholder window is stuck on “Signing in…”

Quit and relaunch the app. The placeholder hydrates from the persisted session on boot — if hydration is partial, the next clean boot usually resolves it.

About

Onereach.ai Lite is the slim companion to the full Onereach.ai desktop app. It started as a strangler-pattern rewrite of the sign-in / settings / IDW surface as a small kernel that ships independently. Both apps share lib/ and follow the same design language.

Source & releases: github.com/wilsr7000/Onereach_Lite_Desktop_App.

For deeper architectural context, every non-trivial decision is recorded as an ADR in lite/DECISIONS.md. Module APIs are documented in the API Reference window (Settings → Developer → Open API Reference).
